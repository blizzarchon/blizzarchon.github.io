Preface

The modding tool is meant for use with FTL data files. If you are unfamiliar with making changes to FTL data files it will be harder to understand the usage of this tool.

If you need some bearings, I encourage you to look through the game files, which you can do from Slipstream Mod Manager via File → Extract Dats...

Motivations

Compatibility

So you've got a weapon mod. It adds a whole bunch of weapons, and as is usually the case, you want to add a bunch of new blue options to give the weapons interesting event applications. The easiest thing to do is create new versions of events and add some blue options. This is fine if you plan on only using this mod, but the problem with replacing events like this is that it doesn't scale well → you run into some problems when you think about compatibility.

The first order of compatibility comes from the source. Now if you're working with vanilla events, this isn't something to worry about, since vanilla is content-complete. If you're working with an overhaul or overhaul addon or something like that, where the events could get updates in future versions, then you might get problems.

Think about it like this. If you replace the original event, and then the original event gets an update, and your mod is actively preventing a feature update!

The second order of compatibility to think about comes from fellow addons. If we both go about this same 'replace the event' method, we may potentially overwrite another mod's changes and then only one of our mods is going to add blue options...

What if instead of replacing the events, we only did the adding of new options? Then we could have mods resilient to update changes and at the same time be compatible with other addons. We can do just that with Slipstream tags.

Possibilities

Slipstream tags can certainly make things less tedious, and tone down how much code is written, while at the same time being more compatible. But what if I told you there are things you can do with them that are just not possible otherwise? Think about this,

So, I'm tired of fire beams not burning everything. I want fire beams to burn every single tile I strike with them. Making a universal thing like this is just not possible with conventional modding. To do it you'd have to find every single mod with a fire beam and create a new one with their fireChance to 10, even people's personal mods, mods with the same name for fire beam but different effects. So yeah, impossible. But this is something we can actually do with Slipstream tags. More on this later.

Terminology

So you've seen some of the reasons we'd want to use the tags. But before jumping in, it's important we define some terms to make sure we're all on the same page here. I'll be referencing these keywords. Take a look at this event: 

			
<event name="MY_EVENT">
	<text>There seems to be nothing at all at this beacon. Time to move on... except, what's that over there?</text>
	<choice>
		<text>Investigate.</text>
		<event>
			<text>There's nothing after all. Bummer.</text>
		</event>
	</choice>
</event>
  • Tag

    'event', 'text' and 'choice' are tags. You can tell as much by the angled brackets '<' and '>'. <event ...> is concluded by the closing tag </event>, containing other tags within. Sometimes the tag can be self-closing like so: <event/>.

  • Value

    The content of a tag. The value of the 'choice' tag on line 3 is: Investigate.

  • Attribute

    name="MY_EVENT" is an attribute - a key-value pair. In this case, 'name' is the key, 'MY_EVENT' the value. The same tag cannot have more than one of the same key, as the keys are unique.

  • Children

    These are the tags directly inside of a tag. For instance, the first 'text' and 'choice' tags are the children of the 'event' tag with name="MY_EVENT".
    Therefore the relationship between the 'choice' on line 3 and the 'text' on line 4 is a parent-child relationship. It may sound a little odd but it will be useful to view the tags in this way.
